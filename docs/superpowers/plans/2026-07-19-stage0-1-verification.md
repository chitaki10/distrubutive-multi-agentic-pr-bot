# Stage 0+1 Vertical Slice — E2E Verification

Date: 2026-07-19

## Setup

- GitHub App `chitaki10-pr-review-bot` (App ID 4337457), installed on `@chitaki10` account, scoped to 4 repos.
- Permissions: `metadata: read` (mandatory), `issues: write`, `pull_requests: write`.
- Webhook: smee.io tunnel (`https://smee.io/FkXILrK2hmrv03`) → `http://localhost:8000/webhook`.
- Model: `qwen2.5-coder:3b` (Q4_K_M) served locally via Ollama, on RTX 3050 Ti Laptop GPU (4GB VRAM).
- Server: `uvicorn prbot.app:app --port 8000`, running against the merged `master` branch (commit `6246cb5`).

## Demo repo

`chitaki10/Job_Recommender_System_with_MCP`

## Result

Confirmed working end-to-end: opening a PR triggered the `pull_request.opened` webhook, which was verified (HMAC signature), fetched the PR diff via the GitHub App installation token, sent it to the local Ollama model, and posted the model's review as a PR comment. GitHub sent an email notification for the bot's comment, confirming delivery.

## Issues hit and fixed during verification

1. **Initial permission gap (403 Forbidden on comment post).** The App was created with `Pull requests: Read-only` + `Issues: Read & write`. Posting a PR comment via `POST /repos/{owner}/{repo}/issues/{pr_number}/comments` requires `pull_requests: write` on the installation, not just `issues: write` — confirmed via `GET /app/installations`, which showed `permissions: {"pull_requests": "read", ...}` even after the App-level permission was changed to Read and write.
2. **Fix required a separate accept step.** Changing the requested permission on the App's Developer Settings page does not retroactively update an existing installation's granted scopes. The installing user had to go to `https://github.com/settings/installations` → the app → accept the pending permission-update request. After that, `GET /app/installations` showed `permissions: {"pull_requests": "write", ...}` with a fresh `updated_at`, and the next PR's comment posted successfully (`200 OK`, no exception).
3. Two earlier test PRs (#1, #2 merged; #3 opened) hit the 403 before the fix was in place — expected, not a code defect. Non-PR events (`closed`) and an unsigned test payload (401) were also observed behaving correctly during diagnosis.

## Conclusion

Stage 0+1 vertical slice deliverable met: a real GitHub PR triggers a real bot-posted review comment, generated by a fully self-hosted open-weight model. No code changes were needed to fix the 403 — it was a GitHub App permission/installation-grant issue, not a bug in `prbot`.
